# 现代商城项目缺口评估

本文档用于记录当前 `uni-mall` 项目相对于一个现代商城项目的主要缺口，分为两部分：

- 业务与产品能力缺口
- 技术栈与基础设施缺口

目的不是否定当前实现，而是明确它目前更接近“可运行的商城骨架 / demo”，距离“可持续运营的现代商城”还差哪些关键层。

## 1. 当前项目现状判断

当前仓库已经具备基础交易闭环：

- 小程序前端页面已覆盖首页、分类、搜索、商品详情、购物车、订单、地址、登录、收藏、优惠券等主要用户页面
- NestJS 后端已提供商品、购物车、地址、优惠券、收藏、订单、登录等 API
- 后端数据库已包含商品、SKU、购物车、地址、优惠券、订单、支付记录等核心表
- 下单逻辑已经具备事务处理、库存扣减、优惠券核销、订单写入、购物车清理等基础能力

但从能力边界看，当前项目仍主要是：

- 面向用户侧的基础商城流程
- 适合演示、联调和继续扩展的项目骨架
- 尚未进入完整商业化、平台化、运营化阶段

## 2. 业务与产品能力缺口

### 2.1 后台运营端缺失

当前仓库只有用户侧前端和业务 API，没有完整后台管理端。现代商城通常至少需要：

- 商品管理
- 类目管理
- SKU 与库存管理
- 订单管理
- 发货管理
- 售后管理
- 优惠券与活动配置
- Banner 和首页装修配置
- 用户管理
- 数据看板

没有后台管理端，商城很难进入真实运营状态。

### 2.2 真实登录与真实支付缺失

当前登录仍是 mock 登录，支付也是模拟支付成功。现代商城上线前通常需要：

- 微信小程序登录
- 短信验证码登录或账号体系补充
- 微信支付下单
- 支付回调验签
- 支付幂等处理
- 退款能力
- 支付异常补偿流程

这是从 demo 到可上线系统的核心门槛。

### 2.3 订单履约链路不完整

当前订单能力主要覆盖下单、支付、取消、确认收货，但仍缺少现代商城常见的履约能力：

- 商家发货
- 物流公司与运单号
- 物流轨迹查询
- 拆单与合单
- 自动关单
- 异常签收处理
- 退款 / 退货 / 换货
- 售后协商记录

没有这部分，订单只能停留在简化状态流转。

### 2.4 商品体系过于简化

当前商品结构足以支撑演示，但对现代商城来说仍然偏轻：

- 缺少主图、详情图、视频等媒体资源体系
- 缺少品牌、属性、参数表、卖点等结构化内容
- SKU 规格能力较弱
- 缺少商品上下架审核
- 缺少预售、限购、阶梯价、组合购等扩展能力
- 缺少商品详情装修能力

现代商城的商品系统通常是整个业务的核心基础设施之一。

### 2.5 营销体系不足

当前只有基础优惠券，缺少更完整的营销玩法：

- 满减 / 满赠
- 秒杀
- 拼团
- 会员价
- 新人礼包
- 积分商城
- 邀请裂变
- 分销
- 活动会场
- 直播 / 内容种草

从增长和转化角度看，这部分是现代商城的重要能力层。

### 2.6 搜索与推荐能力偏初级

当前搜索已具备基础关键字检索，但还缺少：

- 搜索联想
- 热门词配置
- 多维排序
- 筛选条件组合
- 个性化推荐
- 猜你喜欢
- 曝光 / 点击 / 转化埋点
- 基于行为数据的推荐重排

现代商城通常不会只停留在“关键词查询商品列表”。

### 2.7 会员与用户资产体系不足

当前“我的”页已经有积分、余额等展示，但还没有形成真实业务闭环。常见缺失包括：

- 会员等级
- 成长值
- 积分账户
- 余额账户
- 签到任务
- 消息中心
- 发票能力
- 隐私与账号设置
- 注销与风控能力

这部分决定了留存和复购能力。

### 2.8 风控、客服、审计能力缺失

现代商城通常还需要这些支撑能力：

- 风控规则
- 防刷与反作弊
- 敏感操作审计日志
- 在线客服 / 小程序客服
- 订单与支付操作留痕

当前项目在这方面基本还是空白。

## 3. 技术栈与基础设施缺口

### 3.1 Redis 缓存层缺失

当前没有 Redis。现代商城中 Redis 通常用于：

- 验证码
- 登录态扩展
- 热点商品缓存
- 购物车缓存
- 限流
- 防重提交
- 分布式锁
- 活动库存抢购控制

这是最优先补齐的基础设施之一。

### 3.2 消息队列或异步任务栈缺失

当前没有消息队列或任务调度体系。现代商城常见场景包括：

- 支付回调异步处理
- 订单超时自动关闭
- 发货通知
- 短信 / 模板消息发送
- 埋点事件消费
- 库存异步同步
- 退款任务

可选方向通常包括：

- `BullMQ + Redis`
- `RabbitMQ`
- `Kafka`

### 3.3 对象存储缺失

当前商品展示大量依赖文本和静态资源，缺少正式文件存储方案。现代商城通常需要：

- 商品主图
- 商品详情图
- 营销活动图片
- 用户头像
- 运营素材

常见接入包括：

- `阿里云 OSS`
- `腾讯云 COS`
- `AWS S3`

### 3.4 真实认证与外部能力集成栈缺失

当前只有 JWT 和 mock 登录，还缺少：

- 微信小程序登录能力
- 手机验证码服务
- 微信支付 SDK / 接口接入
- 支付回调验签
- 退款接口接入
- 第三方物流接口

这些不是单纯的业务逻辑补充，而是关键技术栈的一部分。

### 3.5 数据库治理能力不足

当前使用原生 SQL 并没有问题，但数据库工程治理还可以继续加强：

- migration 规范化管理
- SQL 审计与回滚策略
- 慢查询监控
- 连接池指标监控
- 读写分离预留
- 索引治理策略

当前更偏“可运行”，还不是“可长期稳定演进”。

### 3.6 后台权限体系缺失

现代商城后台通常至少需要：

- RBAC 角色权限模型
- 操作菜单权限
- 资源级权限控制
- 运营 / 客服 / 仓库 / 管理员角色划分

当前只有用户侧 JWT 鉴权，不足以支撑管理后台。

### 3.7 前端状态管理体系不完整

前端依赖中已有 `pinia`，但目前整体还是偏页面直连 API + 本地缓存，缺少更系统的状态层设计，例如：

- 用户态
- 购物车态
- 全局配置态
- 活动态
- 消息通知态

项目继续变大后，这部分会影响维护成本。

### 3.8 前端业务组件体系不完整

现代商城前端通常会沉淀一层业务组件，而不只是通用 UI 组件：

- SKU 选择弹层
- 价格与营销标签组件
- 地址选择组件
- 订单状态组件
- 优惠券卡片组件
- 活动会场组件

这类组件层是前端可维护性的重要部分。

### 3.9 埋点与数据分析能力缺失

当前没有看到前端或后端埋点体系。现代商城常见需要：

- 页面曝光埋点
- 商品点击埋点
- 加购埋点
- 支付转化埋点
- 搜索行为埋点
- 漏斗分析
- A/B 实验支持

没有数据采集，就很难做增长优化。

### 3.10 自动化测试不足

当前仓库虽然有基础构建脚本，但自动化测试仍较弱：

- 后端缺系统化单元测试
- 后端缺接口集成测试
- 前端缺页面流程测试
- 缺少关键交易链路的回归验证

现代商城如果没有测试体系，后期改动风险会迅速上升。

### 3.11 日志、监控、告警栈缺失

当前缺少完整 observability 能力，建议后续补充：

- 结构化日志
- 错误追踪
- 性能指标采集
- 接口耗时监控
- 告警通知

常见方向包括：

- `Pino` / `Winston`
- `Sentry`
- `Prometheus + Grafana`

### 3.12 CI/CD 与环境治理能力不足

当前缺少持续集成和发布治理能力：

- lint / build / test 自动校验
- migration 校验
- 多环境部署流程
- staging / production 区分
- 密钥与配置管理
- 灰度发布能力

现代项目通常会尽早补这部分。

### 3.13 网关与部署扩展能力不足

虽然已有 `Dockerfile` 和 `docker-compose.yml`，但仍缺少更完整的部署栈：

- 反向代理或 API Gateway
- HTTPS 与域名配置
- 多服务编排
- worker 进程部署
- 自动恢复与健康检查
- 更进一步的 K8s 预留

## 4. 优先级建议

如果按现实开发顺序拆分，建议优先级如下。

### P0：从 demo 走向可上线的必要能力

- 微信登录
- 微信支付
- 支付回调与幂等
- 后台管理端
- 发货与退款基础链路
- Redis
- 日志与错误监控

### P1：从可上线走向可运营

- 营销体系扩展
- 商品内容体系增强
- 物流履约完善
- 权限系统
- 对象存储
- 消息队列 / 异步任务
- 自动化测试

### P2：从可运营走向可增长、可扩展

- 推荐系统
- 个性化搜索
- 会员等级与积分闭环
- 埋点分析
- A/B 实验
- 更完整的 CI/CD 与多环境治理

## 5. 推荐优先补齐的技术栈

如果只从技术栈角度选最值得优先补齐的部分，建议顺序如下：

1. `Redis`
2. `BullMQ` 或其他任务队列
3. 对象存储 `OSS / COS / S3`
4. 微信登录与微信支付接入
5. 日志监控告警栈
6. 自动化测试与 CI/CD

## 6. 在不扩业务前提下建议补齐的现代库与工程工具

如果当前阶段不打算继续扩展商城业务，而是希望在保留现有能力的前提下，把项目做得更现代、更稳、更易维护，那么建议优先补的是工程层能力，而不是新的业务模块。

### 6.1 当前已有但还不算完整的基础能力

当前项目已经具备以下基础：

- 使用 `class-validator` + `ValidationPipe` 做入参校验
- 使用 `Joi` 做环境变量校验
- 使用 Swagger 生成接口文档
- 使用 NestJS 模块化结构组织业务

这说明项目并不是完全缺少基础设施，而是“有基础版，但还没形成完整工程体系”。

### 6.2 建议优先补齐的工程能力

#### 结构化日志

当前项目更适合尽快补齐统一日志能力，建议方向：

- `pino`
- `nestjs-pino`
- 开发环境可配合 `pino-pretty`

建议目标：

- 请求日志统一输出
- 错误日志统一输出
- 关键业务日志可追踪
- 日志中带 `requestId`

#### 全局异常格式统一

建议增加全局异常过滤器，统一错误返回结构。理想状态应至少包含：

- `code`
- `message`
- `details`
- `requestId`
- `timestamp`

这样前端处理和线上排查都会明显更清晰。

#### 请求链路追踪

建议引入请求上下文能力，典型方式包括：

- Node.js `AsyncLocalStorage`
- 配合 `nestjs-pino` 的 request context

目标是让一次请求从 controller 到 service 到数据库日志都能串起来。

#### 自动化测试

当前最值得补的不是 ORM，而是测试。建议至少增加：

- `supertest` 用于接口集成测试
- NestJS 测试模块用于 service/controller 验证
- 针对购物车、优惠券、订单的关键链路回归测试

如果后续想进一步现代化，也可以考虑：

- `vitest`

但不强制，沿用 Nest 现有测试栈也完全可以。

#### Git 提交流程与本地质量门禁

建议补充：

- `husky`
- `lint-staged`
- `commitlint`

目标不是“形式化提交”，而是尽量把 lint、格式检查、基础校验前移到提交前。

### 6.3 数据库访问层的现代化方向

当前项目明确采用：

- PostgreSQL
- 原生 SQL
- 显式事务

这条路线本身没有问题，尤其商城里复杂事务较多，不必为了“现代化”强行切换 ORM。

如果希望保留 SQL 风格，同时提升类型安全和可维护性，更值得考虑的是：

- `slonik`
- `kysely`
- `drizzle`

可按风格理解为：

- `slonik`：适合继续以 SQL 为主，但希望更安全、更规范
- `kysely`：适合希望引入类型友好的 query builder
- `drizzle`：适合兼顾 schema、SQL 和类型约束

### 6.4 对 Prisma 的判断

`Prisma` 不是当前项目的必需项，也未必是最合适的选项。

原因包括：

- 当前项目已经围绕原生 SQL 和显式事务组织实现
- 商城订单、库存、优惠券这类逻辑经常需要精细控制 SQL 与锁
- 如果中途强切 ORM，迁移成本较高，但收益未必成正比

因此更合理的判断是：

- 如果团队就是要全面 ORM 化，可以评估 Prisma
- 如果只是为了“更现代”，不建议把 Prisma 当作当前优先项

### 6.5 更现代的 schema 校验方案

当前的 `class-validator` 已经能满足大部分基础需求，但如果后续希望进一步统一类型与校验模型，可以评估：

- `zod`
- `nestjs-zod`

适用价值主要在于：

- 复杂对象校验表达更自然
- schema 与类型更统一
- 前后端 schema 复用潜力更高

但这不是立刻必须迁移的内容。对于当前项目来说，优先级低于日志、异常、测试。

### 6.6 Swagger 与接口类型联动

当前项目已经有 Swagger，但若想进一步提升前后端协作效率，建议继续补充：

- 更完整的 Swagger 注解
- DTO 字段级说明
- OpenAPI 到前端类型或 client 的生成

可考虑方向：

- `@nestjs/swagger` 继续深化使用
- 后续引入 OpenAPI codegen 工具生成前端接口层

### 6.7 代码结构层面的补强建议

即便不引入新库，当前代码结构也建议逐步补一层边界：

- 将较重的 SQL 查询逐步沉淀为 repository / query 层
- service 更聚焦业务编排
- 避免订单等复杂模块持续膨胀为“大型 service 文件”

这类调整比“为了现代而换库”更实际。

### 6.8 最值得优先引入的现代库清单

如果只选一小批高性价比工具，建议优先考虑：

1. `nestjs-pino`
2. `pino-pretty`
3. `supertest`
4. `husky`
5. `lint-staged`
6. `commitlint`
7. `zod` 或 `nestjs-zod`
8. `kysely` 或 `slonik`

### 6.9 当前不建议优先投入的方向

以下内容不属于当前阶段最值得优先处理的事项：

- 为了“现代”而强行切 `Prisma`
- 过早引入大而全的 DDD 框架
- 过早拆微服务
- 过度事件化、过度抽象基础设施

当前项目更需要的是把已有单体商城骨架做稳，而不是过早追求架构复杂度。

## 7. 结论

当前项目已经具备一个商城项目最基础的用户侧交易能力，但它还缺少现代商城最关键的三层能力：

- 运营层能力
- 生产化基础设施
- 增长与数据能力

后续如果继续推进，建议不要只围绕“再补几个接口”展开，而是按系统层级补齐：

1. 交易闭环生产化
2. 运营后台与履约体系
3. 营销、会员、推荐、数据分析能力

这样项目才会从“演示商城”逐步进入“现代商城”的合理形态。
